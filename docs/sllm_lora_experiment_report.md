# sLLM LoRA/QLoRA 파인튜닝 실험 보고서

## 1. 실험 목적

본 실험은 청년 정책 통합 탐색 에이전트 프로젝트에서 sLLM 기반 파인튜닝 가능성을 확인하기 위해 수행하였다.  
운영 서비스의 메인 구조는 OpenAI 기반 RAG 파이프라인을 유지하되, 평가 기준의 `sLLM 파인튜닝 및 최적화 코드 역량` 항목에 대응하기 위해 LoRA/QLoRA 기반 PEFT 파인튜닝 코드와 샘플 학습 결과물을 구성하였다.

본 실험의 핵심 목적은 다음과 같다.

```text
1. 정책/훈련/창업공고 데이터를 instruction tuning 형식으로 변환할 수 있는지 확인
2. LoRA adapter 학습 및 저장이 정상적으로 수행되는지 확인
3. 학습된 adapter를 base model에 결합하여 응답을 생성할 수 있는지 확인
4. sLLM 단독 응답 방식의 한계와 향후 개선 방향을 도출
```

## 2. 실험 환경 및 구성

| 항목 | 내용 |
|---|---|
| Base Model | `Qwen/Qwen2.5-1.5B-Instruct` |
| Adapter Path | `sllm/outputs/lora_policy_qa` |
| 학습 방식 | LoRA / QLoRA 대응 PEFT 파인튜닝 |
| 데이터 형식 | instruction, input, output 기반 JSONL |
| 테스트 방식 | 학습된 adapter 로드 후 샘플 프롬프트 5개 일괄 생성 |
| 테스트 결과 파일 | `sllm_prompt_test_results.jsonl` |
| 최초 테스트 시각 | `2026-06-15T09:20:01` |

## 3. 학습 및 테스트 절차

실험 절차는 다음과 같다.

```text
기존 전처리 데이터
↓
prepare_sllm_dataset.py
↓
instruction tuning용 JSONL 생성
↓
train_lora.py
↓
LoRA/QLoRA adapter 학습 및 저장
↓
batch_test_sllm_prompts.py
↓
샘플 프롬프트 5개 일괄 생성 테스트
```

학습 데이터는 청년 정책, 창업지원 공고, 교육훈련 과정 정보를 instruction format으로 변환하여 구성하였다.  
모델은 전체 파라미터를 업데이트하지 않고 LoRA adapter만 학습하여 GPU 메모리 사용량을 줄이는 방식으로 구성하였다.

## 5. 주요 관찰 결과

### 5.1 정상 동작한 부분

학습된 LoRA adapter는 base model에 정상적으로 결합되었고, 5개의 테스트 프롬프트에 대해 모두 오류 없이 응답을 생성하였다.  
또한 응답 형식은 대체로 청년 정책 안내, 교육훈련 과정 안내, 창업지원 공고 안내와 같은 형태를 따랐다.

확인된 긍정적 결과는 다음과 같다.

```text
1. LoRA adapter 로드 및 추론 성공
2. 모든 테스트 프롬프트에서 error=null 기록
3. 한국어 응답 생성 가능
4. 청년 정책/지원사업 안내 형식 일부 학습
5. 정책명, 분야, 신청기간, URL, 출처 등 구조화된 표현 일부 생성
```

### 5.2 품질상 한계

응답 품질은 운영 서비스에 바로 적용하기에는 부족하였다.  
특히 sLLM 단독 생성 방식에서는 RAG 검색 결과를 실시간으로 참조하지 않기 때문에 최신성, 출처 정확성, 마감 여부 판단에서 문제가 발생하였다.

대표적인 한계는 다음과 같다.

```text
1. 월세 지원 질문에 K-Startup 공고 URL을 생성하는 등 도메인과 출처가 불일치함
2. 2026년에 신청 가능한 창업공고 질문에 2025년 신청기간을 포함하여 응답함
3. 면접수당 질문에 2019년 정책을 추천하여 오래된 정책 배제에 실패함
4. AI 데이터 분석 훈련 과정 및 금융 정책 설명에서 동일 문장 반복이 발생함
5. 실제 원문 검색 없이 모델이 학습 패턴을 기반으로 URL과 세부 내용을 생성하여 hallucination 위험이 존재함
```

## 6. 운영 RAG 파이프라인과의 비교

본 프로젝트의 운영 서비스는 Vector DB 검색, 마감 상태 필터링, ReAct 기반 충분성 판단, 외부 공식 출처 fallback 계획을 포함한다.  
반면 이번 sLLM 실험은 RAG context 없이 학습된 adapter만으로 응답을 생성하였다.

따라서 두 방식의 역할은 다음과 같이 구분하는 것이 적절하다.

| 구분 | OpenAI 기반 RAG 파이프라인 | sLLM LoRA 실험 모델 |
|---|---|---|
| 목적 | 실제 서비스 응답 생성 | 파인튜닝 가능성 검증 |
| 최신성 | Vector DB 및 마감 상태 필터 기반 판단 가능 | 학습 데이터에 의존하여 최신성 취약 |
| 출처 정확성 | 검색 결과의 source_url 기반 | 잘못된 URL 생성 가능 |
| 마감 공고 배제 | deadline_status, is_expired 기반 처리 가능 | 오래된 정책 생성 가능 |
| 응답 안정성 | 상대적으로 높음 | 반복 생성 및 hallucination 가능 |
| 적합한 활용 | 운영 서비스 메인 응답 | 조건 추출기, 요약기, 초안 생성기 등 보조 모듈 |

## 7. 개선 방향

이번 실험 결과를 바탕으로 다음 개선 방향을 도출하였다.

```text
1. sLLM 단독 QA 방식이 아니라 RAG 검색 context를 input에 포함한 supervised fine-tuning 데이터로 재구성
2. 학습 데이터에서 마감 공고, 오래된 정책, 부정확한 URL을 제거하거나 별도 라벨링
3. 출력 형식을 정책명, 지원내용, 신청기간, 출처, 확인 필요 사항으로 고정
4. repetition_penalty, no_repeat_ngram_size, max_new_tokens 조정을 통해 반복 생성 완화
5. 평가 시 최신성, 출처 정확성, 반복 여부, hallucination 여부를 별도 기준으로 측정
6. 운영 서비스에는 바로 적용하지 않고 조건 추출, 요약, 답변 초안 생성 등 제한된 보조 기능부터 적용 검토
```

## 8. 결론

LoRA/QLoRA 기반 sLLM 파인튜닝 실험은 기술적으로 정상 수행되었다.  
학습 데이터 생성, adapter 학습, adapter 로드, 프롬프트 일괄 테스트까지 완료되었으며, 모든 테스트 케이스에서 오류 없이 한국어 응답이 생성되었다.

다만 응답 품질 측면에서는 최신성 판단, 출처 정확성, 오래된 정책 배제, 반복 생성 억제에서 한계가 확인되었다.  
따라서 본 프로젝트의 운영 서비스에는 기존 OpenAI 기반 RAG 파이프라인을 유지하는 것이 적절하며, sLLM은 향후 RAG context를 입력으로 받는 경량 답변 생성기 또는 조건 추출기/요약기 형태로 확장하는 것이 바람직하다.

최종적으로 본 실험은 sLLM을 운영 모델로 대체하기 위한 결과라기보다, 평가 기준에서 요구하는 PEFT 기반 파인튜닝 코드 역량과 실험적 검증 과정을 보여주는 보조 산출물로 의미가 있다.
